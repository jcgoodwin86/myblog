Here's my full review of the project:

  ---
  Project Review: Go Blog (myblog)

  ---
  Architecture Overview

  net/http (main.go)
    → internal/handler/{home,post}.go
      → internal/model/post.go         (disk read + frontmatter parse)
        → internal/render/markdown.go  (Goldmark instance)
      → templates/pages/*.templ
        → templates/layout/base.templ
        → templates/components/markdown.templ

  The layering is clean and intentional. Good start.

  ---
  Templ Components

  Strengths:
  - Clear three-tier layout: layout/, pages/, components/ — good separation of concerns.
  - base.templ provides the single root HTML shell with { children... } — proper composition pattern.
  - components/markdown.templ isolates templ.Raw() to one place, which is the right call for a potentially dangerous operation.
  - Data flows cleanly: handler → model → page component → layout.

  Issues:

  1. templ.Raw() — XSS risk depending on input (templates/components/markdown.templ:5)
  templ.Raw(string(html)) bypasses Templ's auto-escaping. This is acceptable here because the HTML comes from Goldmark rendering your own trusted Markdown files — not user input. However, Goldmark by default passes raw HTML
  through if Markdown contains <script> tags or inline HTML. If content files ever come from untrusted sources, this becomes a real XSS vector. Consider enabling goldmark.WithRendererOptions(html.WithUnsafe(false)) explicitly
  to document intent.

  2. Error not checked in HandlePost (internal/handler/post.go:21)
  pages.Post(*postData).Render(r.Context(), w)
  The error return from .Render() is silently discarded. HandleHome handles this correctly. This is inconsistent and masks rendering failures.

  3. Home page has no content (templates/pages/home.templ)
  <div>Home</div>
  Obviously a placeholder, but noted for completeness.

  4. No nav.templ or postcard.templ — the README describes these components but they don't exist yet.

  ---
  Goldmark Pipeline

  Strengths:
  - Goldmark instance is centralised in internal/render/markdown.go as a package-level var — correct approach.
  - goldmark-meta extension is properly wired into the parser context flow.

  Issues:

  5. No syntax highlighting (internal/render/markdown.go)
  The README mentions Chroma but no goldmark-highlighting extension is configured. The pipeline is missing it entirely right now.

  6. Goldmark renders on every request (internal/model/post.go:LoadPost)
  Every call to HandlePost hits disk (os.ReadFile) and runs the full Goldmark conversion pipeline. For a blog with infrequently changing content, this is unnecessary overhead. A simple sync.Map or map[string]*Post cache
  populated at startup (or lazily with sync.Once) would eliminate repeated disk reads and parses for the same post.

  7. Goldmark safety not explicitly configured
  Goldmark's default renderer allows raw HTML passthrough from Markdown (e.g., <script> in .md files renders as-is). The current code has no html.WithUnsafe(false) / html.WithXSSFilter configuration. Since content is currently
  trusted, this is low risk, but worth being explicit about.

  ---
  net/http → Handler → Model Data Flow

  Strengths:
  - Handlers are appropriately thin — they fetch data and render, nothing more.
  - Slug sanitization is present (strings.ContainsAny(slug, "./\\")).
  - Proper HTTP timeouts set in main.go.
  - Pattern-based routing with GET /posts/{slug} uses Go 1.22+ routing correctly.

  Issues:

  8. Slug sanitization is incomplete (internal/handler/post.go:10-13)
  if strings.ContainsAny(slug, "./\\") {
  This blocks obvious path traversal like ../../etc/passwd, but filepath.Join("content/posts", slug+".md") already cleans the path. A more robust guard would be to verify the resolved path starts with the expected base
  directory:
  base := filepath.Clean("content/posts")
  resolved := filepath.Join(base, slug+".md")
  if !strings.HasPrefix(resolved, base+string(os.PathSeparator)) {
      http.NotFound(w, r)
      return
  }

  9. 404 vs 500 conflation (internal/handler/post.go:18-21)
  If os.ReadFile fails because the file doesn't exist (slug not found), the handler returns 500 Internal Server Error. It should check for os.IsNotExist(err) and return http.NotFound instead. Currently, a request for
  /posts/doesnt-exist returns 500.

  10. content/posts path is hardcoded and relative (internal/model/post.go:15)
  path := filepath.Join("content/posts", slug+".md")
  This path is relative to the process working directory. If the binary is run from a directory other than the project root, it breaks silently. Consider using os.Executable() or an injected base path.

  11. postFromMeta swallows the error for missing date (internal/model/post.go:53-59)
  If the date field is missing or unparseable, the function returns nil, err — but the caller ignores the second return value:
  postData, _ := postFromMeta(metaData) // error discarded
  A post with a malformed or absent date silently gets the zero time.Time value rather than returning an error to the client.

  ---
  Markdown Content / Frontmatter

  The single sample file (content/posts/firstpost.md):
  ---
  title: "My First Post"
  date: 2026-02-28
  tags: ["go", "web"]
  ---

  Observations:
  - Frontmatter structure is sensible: title, date, tags — all mapped in the model.
  - description is defined in the Post struct but absent from the sample frontmatter.
  - Tags in the frontmatter are a JSON-style array (["go", "web"]), but goldmark-meta parses frontmatter as YAML. In YAML, ["go", "web"] is valid inline sequence syntax, so this works — but the more idiomatic YAML form would
  be:
  tags:
    - go
    - web
  - There's no slug in the frontmatter — the slug is derived from the filename, which is a clean approach.
  - No draft field — something to consider for not accidentally serving unfinished posts.
  - Only one post exists; directory is flat under content/posts/. Fine for now but no category/section structure yet.

  ---
  Module / Dependencies

  12. Go version in go.mod is go 1.25.0
  As of the current date (March 2026) this is plausible, but worth confirming this is the actual toolchain version rather than a typo for 1.23 or 1.24.

  13. go.work.sum in .gitignore but no go.work
  The .gitignore excludes go.work.sum, implying a Go workspace might be used locally. Fine — just noting the implication.

  14. README mentions HTMX but no HTMX is present in the code or dependencies. The static/js/ directory is empty. The README appears to describe a planned architecture rather than the current one.
